## Зачем нужен проект
Проект нужен для централизованного сбора инфраструктурных метрик и логов приложений на одном сервере.

`node-exporter` отдает системные метрики хоста для мониторинга (CPU, RAM, FS, сеть), а `vector-agent` читает логи приложений (`pm2`, `nginx`, `mysql`), нормализует их в единый JSON-вид и складывает в каталог для дальнейшего анализа.


## Node-Exporter
**Стек (образ):** `prom/node-exporter:v1.10.2`.
**Пользователь запуска (prod):** `nobody` (`uid: 65534`, `gid: 65534`).

**Блок `node-exporter`:** контейнер снимает метрики напрямую с хоста через `volumes` (`/proc`, `/sys`, `/` в `ro`), поэтому выбран режим `read_only` и `cap_drop: ALL`.

Такое решение нужно, чтобы сервис имел доступ только на чтение к данным хоста и не мог менять состояние системы.

Порт опубликован как `127.0.0.1:9100:9100`, потому что метрики должны быть доступны локально для внутреннего стека мониторинга, а не извне.

Сеть `monitoring_net` используется отдельно, чтобы изолировать канал метрик от канала логов.


## Vector
**Стек (образ):** `timberio/vector:0.51.1-alpine`.
**Пользователь запуска (prod):** `uid: 2000`, `gid: 2000`.
**Дополнительная группа:** `adm`.

Так как логи монтируются с хоста (`${PM2_LOGS_DIR}`, `${NGINX_LOGS_DIR}`, `${MYSQL_LOGS_DIR}`), на хосте должны быть корректно выставлены права доступа для группы `adm` (минимум право чтения), иначе `vector-agent` не сможет читать файлы логов.

Каталог `${LOGS_OUT_HOST}:${LOGS_OUT_CONTAINER}` нужен для сохранения уже обработанных логов на хосте вне контейнера, а `./vector_data:/var/lib/vector` сохраняет внутреннее состояние Vector между перезапусками (чекпойнты чтения файлов).

`cap_drop: ALL` и `security_opt: no-new-privileges:true` добавлены для минимизации привилегий процесса.


## Конфигурация-Vector
**Блок `sources`:** подключает 5 источников файловых логов (`pm2`, `nginx`, `mysql error/general/slow`) и читает их с `read_from: end`.

Это сделано специально, чтобы при старте не перечитывать исторический архив, а забирать только новые события.

**Блок `transforms`:** нормализует формат событий. Для `pm2` сообщение парсится как JSON (`parse_json!`), для `nginx/mysql` формируется единая структура с полями `timestamp`, `service`, `log_type`, `file`, `message`.

Такой подход нужен для единообразия данных: дальше логи проще фильтровать, агрегировать и отправлять в другие системы.

**Блок `sinks`:** сохраняет результаты в файловую иерархию по часам (`%Y/%m/%d/%H.log`) внутри `${LOGS_OUT_CONTAINER}`.

## Сети-в-docker-compose.prod.yml
`monitoring_net` помечена как `external: true`.

Это сделано, чтобы сеть можно было переиспользовать в нескольких compose-проектах. Поэтому ее нужно создать заранее до запуска `docker-compose.prod.yml`.

`logs_net` остается внутренней сетью текущего compose для обмена внутри логового контура.


## Переменные-env
`PM2_LOGS_DIR` - путь к логам PM2 на хосте; используется в `vector-agent` как источник `pm2_logs`.
`MYSQL_LOGS_DIR` - путь к логам MySQL; используется для `mysql_error_logs`, `mysql_general_logs`, `mysql_slow_logs`.
`NGINX_LOGS_DIR` - путь к логам Nginx; используется для источника `nginx_logs`.
`VECTOR_CONFIG_DIR_HOST` - путь к каталогу с `vector.yaml` на хосте; монтируется в `/etc/vector`.
`LOGS_OUT_CONTAINER` - путь внутри контейнера Vector, куда пишут `sinks`.
`LOGS_OUT_HOST` - путь на хосте, куда сохраняются обработанные логи из `LOGS_OUT_CONTAINER`.
